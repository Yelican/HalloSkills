# HalloChatGold — 对话淘金术

**把任何讨论变成认知资产。** 基于 6 个被验证的思维模型的有机结合，形成一套 7 步流水线，让每次对话都能快速提炼出洞察、行动和长期价值。

> 想了解这 6 个模型是如何被验证、筛选、最终融合成这套方法的？ → [BIRTH.md](./BIRTH.md)

## 底层模型溯源

| 步骤 | 模型名称 | 学术/业界来源 | 验证状态 |
|------|----------|-------------|---------|
| 1. 收集 | 原始对话文本 | — | 基础操作 |
| 2. 过滤 | 4层价值漏斗 | DIKW金字塔 (Ackoff 1989) | 60+年验证，经管/信息科学标准框架 |
| 3. 评估 | 5级ROI评估 | Kirkpatrick四级 (1959) + Phillips ROI (1996) | 培训评估金标准，学术引用量过万 |
| 4. 提炼 | 3级洞察金字塔 | Peirce符号学 → 批判性思维层级 | 100+年哲学/认知科学验证 |
| 5. 验证 | 反事实价值模型 | Pearl反事实因果推理 (《The Book of Why》2018) | 图灵奖得主方法论，经济学/政策评估标准工具 |
| 6. 落地 | RACI行动矩阵 | PMI项目管理标准 (1960s起) | 项目管理全球标准 |
| 7. 预测 | 复利效应评估 | Naval杠杆率 + 知识半衰期 (T. Forte等) | 3个指标各有独立来源，经创投/知识管理领域验证 |

## 触发条件

### 宽松触发原则

**任何涉及"总结 / 提炼 / 复盘 / 整理 / 结构化"对话内容的请求，都应触发本 Skill。** 宁可多触发一次，不可漏掉。用户不需要说完整的关键词，只要有"把刚才聊的整理一下""这段对话有什么收获""帮我看看我们聊了什么"之类意图即可。

### 必触发

| 用户问法 | 触发动作 |
|---------|---------|
| "帮我总结xxx"、"提炼一下xxx"、"整理xxx"、"复盘xxx" | 走对应管道 |
| "HalloChatGold"、"对话淘金"、"来个复盘" | 完整 7 步管道 |
| "xxx有什么洞见"、"xxx有什么价值"、"xxx值不值" | 精简输出 |
| 用户贴了一大段聊天记录/截图/文本 | 完整 7 步管道 |
| 一段长讨论后用户表现出"收获很大""学到了""原来如此" | 主动提议提炼 |
| 用户说"感觉聊了很多但记不住""信息量好大" | 主动提议提炼 |

### 触发关键词速查

只要用户输入中包含以下任一模式，即为触发：

| 模式 | 示例 |
|------|------|
| 总结 + 对话/聊天/讨论/上下文 | "总结一下我们今天的聊天" |
| 提炼 + 任何内容 | "提炼这段对话的要点" |
| 复盘 | "来个复盘" "复盘一下" |
| 整理 + 对话/聊天/讨论 | "帮我把聊天记录整理一下" |
| 洞见/洞察 + 对话 | "刚才的讨论有什么洞见" |
| 结构化 + 对话/聊天 | "把这段聊天结构化" |
| HalloChatGold | 直接触发 |
| 对话 + 价值/收获 | "这次对话有什么收获" |

### 隐性触发（场景感知）
- 用户发了一段文本/截图后问"怎么样""怎么看""有什么想法"
- 用户说"记了很多但不知道怎么整理"
- 用户说"感觉聊了很多但记不住"
- 上下文刚经历一段深度讨论，用户表现出满足/收获感
- 用户连续多次追问同一个话题，表现出想"吃透"的意图

### 支持场景
- 和 AI（Claude/ChatGPT/DeepSeek）的深度讨论
- 微信/QQ/飞书和别人聊天的文本
- 线下交流后的回忆/笔记
- 读书笔记/听课笔记的提炼
- 任何需要"从对话里榨价值"的场景
- **当前会话上下文本身就是讨论——用户说"总结刚才聊的"就是触发**

## 7 步管道详解

### 第 1 步 · 收集（Collect）— 原始材料

**如果用户提供文本/截图：** 直接读取并分段。
**如果是刚发生的对话：** 基于当前会话上下文回顾主要内容。
**如果是外部聊天：** 让用户粘贴文本（微信截图需要用户先转文字）。

输出：讨论主题 + 参与方 + 时长/篇幅概要。

---

### 第 2 步 · 过滤（Filter）— 4层价值漏斗

> 来源：DIKW金字塔 (Ackoff 1989)。90% 的讨论价值集中在最上层的 10% 内容里。

**内容分层就是帮你分清「哪些值得反复看，哪些扫一眼就够」。**
一次对话的内容可以拆成四层，越往上越值钱——底层是操作，顶层是认知。

| 层级 | 定义 | 讨论中的对应内容 | 占比参考 |
|------|------|----------------|---------|
| **L1 事实信息** | 客观的、可查的操作性知识 | 命令、链接、配置参数、具体数字 | ~10% |
| **L2 方法技能** | 解决某一类问题的通用步骤 | 工作流、配置方法、操作流程 | ~20% |
| **L3 能力迁移** | 能用到其他领域的通用能力 | 工具选型方法论、决策框架 | ~30% |
| **L4 底层认知** | 改变决策方式的底层规律 | "集成度与能力成反比"类规律 | ~40% |

**操作：**
- 把讨论内容逐条扔进对应层级
- L1 只保留关键链接和命令（放附录）
- **重点整理 L3 和 L4**

**分层结论怎么看：**
- L3+L4 占比越高 → 这次讨论越有深度，值得反复消化
- L1 占比高 → 偏操作型，实用但保质期短，提取关键信息就走

---

### 第 3 步 · 评估（Evaluate）— Kirkpatrick 五级评估

> 来源：Kirkpatrick 四级 (1959) + Phillips ROI (1996)。评估单次讨论的投入产出比。

**五级评估从「当时爽不爽」一直追问到「长期值不值」，给你的讨论打个分。**

| 级别 | 评估维度 | 评分(0-2) | 判定标准 |
|------|---------|-----------|---------|
| **L1 反应** | 参与感和满意度 | — | 0=被动听讲 1=有参与但不深 2=沉浸式深度交流 |
| **L2 学习** | 知识/认知增量 | — | 0=没有新东西 1=学到具体知识 2=刷新了底层认知 |
| **L3 行为** | 可迁移到其他场景 | — | 0=用不上 1=可复用特定场景 2=可跨领域迁移 |
| **L4 成果** | 有没有产生实际产出 | — | 0=没有 1=有行动方向 2=有明确可执行的行动清单 |
| **L5 ROI** | 长期回报率 | — | 0=一次性消费 1=影响近期决策 2=影响长期决策 |

**总分 / 10，判定：**
- 8-10：高价值讨论，值得详细提炼
- 5-7：中等价值，快速过一遍
- 1-4：低价值，以后少花时间

**评估结论怎么看：**
- 总分决定整体价值，但**单项低分才是关键** — 那个维度就是下次讨论可以优化的方向
- 比如 L3 行为分低 → 下次聊完直接问自己「这个我能用在哪儿」
- L5 ROI 分低 → 这次偏消费性聊天，下次带着问题聊

---

### 第 4 步 · 提炼（Extract）— 3级洞察金字塔

> 来源：Peirce 符号学 → 批判性思维层级。事实会过时，观点有局限，洞察永不过时。

**操作流程：**

1. **Facts 提取** — 列出讨论中所有客观可验证的信息点
2. **Opinions 提取** — 从 Facts 中输出主观判断和经验总结
3. **Insights 提炼** — 追问："这个观点背后的底层规律是什么？它还能解释什么其他现象？"

**输出格式：**

> ### 💎 核心洞察
>
> **洞察 1：[一句话]**
> - 来源事实：[哪些讨论内容支撑这个洞察]
> - 解释力：[为什么你考研能坚持但健身不能？还能解释什么其他现象]
> - 行动推演：[如果要验证这个洞察，明天可以做什么具体动作]
> - 关联挑战：[这个洞察和「XX」矛盾吗？什么情况下它不成立]
>
> **洞察 2：...**

**洞见质量标准：**
- 上限 7 条，不设下限，**可少不可滥** — 宁可只有 2 条精的，不要硬凑 7 条水的
- 必须经过三轮筛查才能入选：① 有事实支撑？② 能解释 2 个以上无关现象？③ 有可证伪性？
- 每条洞察必须能展开 200 字以上的实质性思考（不够展开的就合并或删掉）
- 重点不是"对不对"，是"有没有用" — 能不能指导行动或改变认知

**洞察结论怎么看：**
- 好的洞察每条都值得你停下来想 5 分钟——它解释了过去，也指向了未来
- 如果看完觉得「有道理但不知道然后呢」→ 说明这条洞察还没挖到底，继续追问「那我能做什么」

**金句收尾：**
所有洞察输出完后，追加一句与主题高度相关的名言/格言，并简短联系本次核心洞察做延展。不用解释金句本身，而是用它照亮讨论中的一个侧面。宁缺毋滥，没有合适的就不加。

---

### 第 5 步 · 验证（Verify）— 反事实价值模型

> 来源：Pearl 反事实因果推理 (2018)。避免"理所当然"错觉，真正测量讨论的不可替代性。

**反事实验证就是问自己：如果没聊这次，我会亏多少？**
听起来扎心，但只有这么问才能挤出讨论的真实价值。

**灵魂三问：**

| 问题 | 用意 |
|------|------|
| "如果今天没有这次讨论，我会怎么做？" | 测量替代路径的成本 |
| "如果今天没有这次讨论，我会损失什么？" | 量化讨论的净价值 |
| "为了获得这次讨论的价值，我愿意付多少钱？" | 把模糊价值转成具体数字 |

**输出：** 一个反事实判断——"这次讨论替我节省了 X 天摸索时间，避免了 Y 个坑，价值至少 Z 元。"

**验证结论怎么看：**
- **节省了时间** → 这次讨论帮你走了捷径
- **避免了坑** → 价值比表面数字更大（没发生的损失最难被感知）
- **愿意付钱** → 绝对值越高，说明这次讨论对你越不可替代

---

### 第 6 步 · 落地（Act）— RACI 行动矩阵

> 来源：PMI 项目管理标准 (1960s起)。没转化的讨论价值为 0。

**RACI 是个责任分配工具，帮你搞清楚：谁干活、谁背锅、问谁意见、告诉谁结果。**
一次高质量讨论的价值，最终体现在你接下来做的事上。

| 角色 | 定义 | 行动项 | 截止时间 |
|------|------|--------|---------|
| **R（执行者）** | 具体干活的人（就是你） | [具体行动] | [时间] |
| **A（问责人）** | 对最终结果负责的人（通常也是你） | — | — |
| **C（顾问）** | 需要问一嘴意见的人/资源 | — | — |
| **I（告知对象）** | 需要知会一声的人 | — | — |

> 标准 PMI 定义：R = 干活的人，A = 对最终交付质量负责的人（通常 R 和 A 是同一个人），C = 提供输入的人，I = 只需知道结果的人。A 不是"批准"，是"问责"。

**所以这次讨论之后：**
1. 先把 R 列的事排进日程
2. 如果需要别人配合 → 填 C（什么时候问谁）
3. 如果有需要同步的人 → 填 I（什么时候告知谁）
4. 至少 1 个 R 类行动，每个必须有明确截止时间

---

### 第 7 步 · 预测（Forecast）— 复利效应评估

> 来源：Naval 杠杆率 + Tiago Forte 知识半衰期 + 网络效应衍生价值。

**复利效应，简单说就是：这次讨论的价值是「用完就废」还是「越滚越大」？**

三个指标决定这次讨论在你生命里的「复利系数」：

| 指标 | 定义 | 本次评分(1-10) | 说明 |
|------|------|---------------|------|
| **杠杆率** | 学会 1 个方法能解决多少类似问题 | — | 1:1（一次性）→ 1:100（爆了） |
| **半衰期** | 这次讨论的价值能持续多久 | — | <月/月-年/>5年 |
| **衍生价值** | 能带来多少其他机会 | — | 效率提升/知识传播/人脉连接 |

**综合复利评级：**
- S 级：杠杆率 ≥1:50 + 半衰期 ≥1年 + 衍生价值明确 → "改变轨迹的对话"
- A 级：3 项中满足 2 项
- B 级：3 项中满足 1 项
- C 级：都不满足（但至少此刻有用）

**评级留言：**
评级不只是打分，是告诉你该怎么对待这次讨论的成果——

| 评级 | 它意味着 | 你该怎么做 |
|------|---------|-----------|
| S | 这次讨论可能改变你的轨迹 | 整理成笔记，纳入核心框架，定期回顾 |
| A | 长期价值明确，值得投入 | 一周内把行动落地，让复利转起来 |
| B | 有收获但深度不够 | 下次围绕同一话题再聊一次，挖深一层 |
| C | 当下有用，但别当宝贝 | 提取可用信息就走，别花时间反复看 |


## 应用示例

**输入：** 一段关于"Claude Code + MCP 搭建论文写作工作流"的讨论

**输出片段（第4步示例）：**

> ### 💎 核心洞察
>
> **洞察 1：所有上层 AI 产品都是底层 API 的阉割版**
> - 来源：Cursor/VS Code 集成 vs 原生 API 的能力差异
> - 解释力：也解释了为什么 APP 不如网页版功能全、为什么中间件总有延迟
> - 行动推演：下次选工具前先问「这个产品的封装值得我失去多少灵活性」
> - 关联挑战：当产品调度能力远超个人工作流时（比如企业级编排平台），这个洞察不成立
>
> **洞察 2：易用性永远以牺牲灵活性为代价**
> - 来源：MCP 配置需要手写 JSON，但换来最大可定制性
> - 解释力：解释了开源 vs 闭源、GUI vs CLI、DIY vs 一站式等各种 tradeoff
> - 行动推演：做技术选型时把「未来可能需要的灵活性」列进评估表
> - 关联挑战：如果用户永远不需要灵活性（纯消费场景），这个 tradeoff 不成立
>
> 📜 就像 Alan Kay 说的：「The best way to predict the future is to invent it.」 这次讨论的核心不是选什么工具，而是你愿意花多少自由度去搭建属于自己的未来。

## 执行原则

1. **不装腔** — 洞察必须能在现实中验证，不说"赋能""底层逻辑"之类的空词
2. **洞察上限 7 条，但核心是「可少不可滥」** — 每次入选都要经过"有事实吗→能解释别的吗→有反例吗"三轮筛查，凑不齐就别硬凑
3. **每条行动必须有截止时间** — 没有时间 = 不会做
4. **反事实必须具体** — 不能是"我会更迷茫"这种废话，要说出具体替代路径
5. **用户说够了就停** — 别为了走完 7 步而走 7 步

## 视角规则（重要）

这份报告是给**讨论的参与者本人**看的，不是写给别人看的第三者报告。

- **正文全程用「我」的视角**，或者按中文习惯省略主语
- **禁止用「用户」称呼使用者**，也不要用 AI 的人称来写这篇报告
- 报告应该像是参与者自己复盘写的，不是 AI 在评价他
- 如果 AI 有人设（取了名字），可以在文末加一行 `编辑：XXX / AI 辅助生成`，但正文不能变成 AI 视角

**正例：**
```
洞察：我真正缺的不是工具，是决定用工具把事做完的耐心
```

**反例：**
```
洞察：用户真正缺的不是工具，是决定用工具把事做完的耐心
洞察：阿豪真正缺的不是工具，是决定用工具把事做完的耐心
```

## 导出规则

报告生成后，主动询问使用者是否需要导出。

**支持格式：**
- Word（.docx）— 适合编辑和分享
- PDF — 适合存档和打印
- 导入已接入的笔记软件（Obsidian / 思源笔记等）

**文件命名规则：**
```
{日期}_{对话对象}.{后缀}
```

| 场景 | 示例 |
|------|------|
| 单次讨论 | `2026.5.13_我与王小明.docx` |
| 跨天讨论 | `2026.5.10-5.13_我与王小明.pdf` |
| 群聊讨论 | `2026.5.13_产品群聊复盘.pdf` |

日期格式统一用点号分隔，不加前导零。
